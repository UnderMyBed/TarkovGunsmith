# AI Workflow — Tier C Upgrade Playbook

This document describes the next tier of AI-driven development for this project. **It is not active.** It activates when the user (Matt) decides the project has earned the additional infrastructure cost.

Tier C is "AI-orchestrated": agents do triage, scheduling, schema-watching, and parallel feature work. Humans still approve PRs but no longer run every step manually.

## Activation checklist

Each item is one PR. Do them in order. After each, the project still works at Tier B + that one feature — so you can stop at any step.

## Step 1 — Issue-driven dev

**Files added:**

- `.github/ISSUE_TEMPLATE/bug.yml` — structured bug report (steps to reproduce, expected, actual, env)
- `.github/ISSUE_TEMPLATE/feature.yml` — feature request (problem, proposal, alternatives, success criteria)
- `.github/ISSUE_TEMPLATE/chore.yml` — chore (what, why, blocking?)
- `.claude/skills/triage-issue.md` — skill that:
  1. Reads a GitHub issue (`gh issue view N`)
  2. Decides: brainstorm needed? straight to plan? duplicate?
  3. If brainstorm: invokes `superpowers:brainstorming`
  4. If straight to plan: invokes `superpowers:writing-plans`
  5. Creates a worktree, branch, opens draft PR linked to the issue
  6. Hands off to `superpowers:executing-plans`

**Outcome:** `gh issue create` is the new "start of work."

## Step 2 — Schema watcher

**File added:** `.github/workflows/schema-watch.yml`

```yaml
on:
  schedule:
    - cron: "0 6 * * *" # 06:00 UTC daily
  workflow_dispatch:
jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pnpm install
      - run: pnpm --filter @tarkov/types codegen
      - id: diff
        run: |
          if git diff --quiet packages/tarkov-types/generated; then
            echo "changed=false" >> $GITHUB_OUTPUT
          else
            echo "changed=true" >> $GITHUB_OUTPUT
          fi
      - if: steps.diff.outputs.changed == 'true'
        uses: peter-evans/create-pull-request@v6
        with:
          commit-message: "chore(types): regen tarkov-api schema"
          branch: chore/schema-regen
          title: "chore(types): tarkov-api schema changed"
          body: |
            Auto-generated by schema-watch workflow.
            Diff in `packages/tarkov-types/generated/`.
            Reviewer: check breaking changes vs. queries in `packages/tarkov-data/`.
```

**Outcome:** schema drift becomes a passive notification, not a manual chore.

## Step 3 — Fixture refresh

Same shape as step 2 but weekly, refreshes Vitest snapshot fixtures so tests stay aligned with real API shapes. PR body includes summary of what changed.

## Step 4 — Dependency bumper

A workflow that nightly runs `pnpm outdated`, picks one update at a time, runs `pnpm update <pkg>`, runs all tests, and:

- If green: opens PR with a Claude-generated body summarizing the changelog highlights and noting any deprecation warnings observed in test output.
- If red: opens PR as draft with the failing test logs and a Claude-written hypothesis on the cause.

**Why not Renovate/Dependabot directly?** Because the value-add is the _contextual changelog summary_ and the _hypothesis on failures_, not the bumping itself.

## Step 5 — Parallel dispatch recipes

`docs/ai-workflow/recipes/` containing markdown templates for common cross-cutting features. Each recipe:

1. Names the agents to dispatch in parallel.
2. Defines the contract (shared type or interface) each agent must satisfy.
3. Names the integration step (single agent, sequential, after parallel work completes).

Example recipe: **"Add a new data-driven feature route"** → dispatch `tarkov-data` updater + `web` route scaffolder + `data-proxy` cache-rule writer in parallel; integrate via shared types in `packages/tarkov-types`.

## Step 6 — Module-owner agents

Three long-lived agent definitions:

- `agents/builder-owner.md` — knows everything about `/builder`, the Weapon Builder route, its state machine, its tests, and `weaponSpec()`.
- `agents/matrix-owner.md` — owns `/matrix` and `armorEffectiveness()`.
- `agents/ballistics-owner.md` — owns `packages/ballistics`, the math, and the fixture set.

Each agent definition includes:

- The exact files it owns (path globs).
- The conventions specific to that module.
- A "what's in scope vs. delegated" decision tree.
- Links to the relevant ADRs, specs, and tests.

You invoke them by name when working in their domain. They read their own definition + context, then act.

**Risk:** these agents drift from reality if not maintained. Mitigation: post-merge agent (step 7) updates them on relevant changes.

## Step 7 — Post-merge agents

GitHub Actions on `push: branches: [main]`:

- **`docs-updater`** — checks if the merged PR touched user-facing behavior; if so, edits `apps/web/README.md` features section.
- **`adr-drafter`** — checks if the merged PR introduces a new architectural pattern; if so, drafts a new ADR and opens a follow-up PR.
- **`deploy-verifier`** — waits for CF Pages deployment, then runs a Playwright smoke test against the live URL and posts result to the merge commit as a check.

## What Tier C does NOT do

- It does **not** auto-merge PRs. Humans still approve.
- It does **not** make architectural decisions. Those still go through brainstorming → spec → ADR.
- It does **not** delete or rewrite history. Same git safety rules as Tier B.
- It does **not** spend money. All Tier C infra runs on GitHub Actions free tier + Cloudflare free tier + Anthropic API costs you authorize per-workflow.

## When to consider Tier D

If you ever want truly autonomous feature development (issues self-triage, plans self-approve, PRs self-merge after CI), that's Tier D and would warrant its own design doc. Don't do it without one.
